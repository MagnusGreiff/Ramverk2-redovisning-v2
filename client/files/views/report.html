Kmom01

Berätta utförligt om din syn på Express och Pug och hur de samverkar. Relatera till andra ramverk du jobbat med. Lyft fram de fördelar och eventuellt nackdelar du finner, eller bara notera skillnader och likheter.

Jag tycker om Express dels för att det är Javascript och dels för att man snabbt kan komma igång och arbeta. Jag har inte använd Express tidigare men jag kan se fördelarna med det. Om man vill göra ett eget REST API så är det väldigt enkelt att göra i Express. Man skriver sina routes och skriver ut något så är man klar. Pug är något nytt och jag har inte använt det innan. Jag tycker dock inte om Pug. Jag testade att använda det när jag gjorde övningen men det fastande inte för mig. Jag tycker inte om sätter hur man skriver html:en. Det var just därför jag valde att använda mig av React som frontend istället. Jag har inte jobbat i många olika ramverk innan men ett jag har jobbat i är Anax. Den stora skillnaden mellan Express och Anax som jag har märkt är hur man sätter upp saker. I anax måste man konfigurera de moduler man vill använda sig av i t. ex di.php samt lägga till routes m.m. men i Express så ligger de i node_modules och man kan bara göra import för att få tag på dem.

Berätta om din katalogstruktur och hur du organiserade din kod, hur tänkte du?

Jag har valt att lägga alla mina vyer (.jsx filer) i en egen mapp(views). Alla mina routes (bara 1 just nu) finns i mappen routes. Allt statiskt hamnar i mappen public såsom stylesheet och bilder. Jag har en css mapp som ligger i root och den finns där för att jag använder less och den behövde ligga där. Huvudfilen heter app.js och ligger i root.

Använde du någon form av scaffolding som Express erbjuder?

När jag skapade filerna som finns i me/Redovisa så använde jag inte scaffolding. När jag testade att använda express i me/kmom01 så använde jag scaffolding. Det var mest för att testa hur det fungerade. Det var inte det enklaste att lära sig. Kommandot tycker jag inte är särskilt bra och dokumentationen var inte heller bra.

Jobbar du med Markdown för innehållet, eller annat liknande?

Nej, det gjort jag inte. Jag valde att skriva vanlig html i mina React filer(jsx). Av någon anledning tycker jag inte direkt om markdown. Det känns som om det begränsar det jag vill göra när jag skriver text och vill styla det jag har skrivit.

Kmom02

Har du jobbat med Docker eller andra virtualiseringstekniker innan?

Nej, det har jag inte gjort. Det tror jag inte iallafall, eller om man räknar Virtualbox som det så har jag gjort det. Det ska bli roligt att få lära sig mer om Docker. Nu när jag har lärt mig att jobba med Docker valde jag att byta ut XAMPP mot en docker container som kör PHP Apache. Dels för att jag ville leka rundor med Docker och dels för att jag nu kan testa min kod i olika version av PHP.

Hur ser du på möjligheterna att använda dig av Docker för att jobba med tester av ditt repo?

Att man kan testa flera olika versioner av ett verktyg (node, PHP etc) är viktigt och med Docker kan man göra det på ett simpelt sätt. För att effektivisera mina användning av Docker valde jag att lägga till några Alias i min bash_profile som sköter några olika Dockerkommandon som docker-compose up och docker-compose down m.m. Detta gör att det går snabbare för mig att testa min kod samt behöver jag inte komma ihåg alla kommandon som Docker har utan istället komma ihåg mina Aliases.

Gick allt smidigt eller stötte du på problem?

När jag skulle fixa docker för PHP gick det smidigt. Jag följde övningen och det gick bra, övningen gick igenom allt som jag behövde för att starta upp mina egna Docker kontainrar. När jag skulle fixa docker för node blev det lite problematiskt. Jag visste inte hur jag skulle få det att fungera. Efter lite googling kom jag till någon Github som visade exakt hur man gjorde för att få Docker att fungera med node.

Skapade du din egen image, berätta om den?

Nej, det gjorde jag inte. Jag valde att vänta till nästa kursmoment då vi kommer gå igenom hur man gör. Det ser inte så svårt ut att göra egna images men som sagt så väntar jag.

Kmom03

Berätta vilka tekniker/verktyg du valde för enhetstester och kodtäckning och varför?

Jag valde att använda mig av Scrutinizer för kodtäckning. Jag valde Scrutinizer för att när jag försökte implementera Coveralls och Codecov fungerade det inte. Jag försökte många timmar med att få de att fungera men de vill inte. Scrutinizer hade jag också använt tidigare så det var inga problem att fixa. För enhetstester valde jag att använda mig av Mocha. Vi gick igenom det i övningen och jag fastnade på hur man skrev dem. Strukturen ser väldigt annorlunda ut från vad jag har sätt tidigare och jag tycker verkligen om strukturen. Just nu finns det knappt några enhetstester för att jag inte ha så mycket kod att testa.

Berätta om cin CI-kedja och reflektera över de valen du gjorde?

Jag använder mig av Travis som byggverktyg det är för att jag har använt det tidigare och jag tycker om hur det är strukturerat. I config filen kör jag `make install` samt `make test`. För kodtäckning och kodkvalitet använde jag mig av Scrutinizer samt Codeclimate. Det är av samma anledning som jag valde dessa. Det är för att jag har använt dem innan och vet hur det fungerar. Som jag sade i förra frågan så fick jag inte Coveralls och Codecov att fungera så det var också en av anledningarna.

Reflektera över hur det gick att integrera enhetstesterna i olika Docker-kontainerns och om du ser någon nytta med detta.

Det var lite problematiskt att fixa detta. Jag satt många timmar med att få rätt på detta. Men tillslut fick jag rätt på det. Det är kanske inte den optimala lösningen men det fungerar så jag låter det vara för tillfället. När Docker-kontainern startar så kör den `make clean-all` för att radera alla node_modules som kommer till kontainern, `make setup-tools-js` för att installera alla valideringsverktyg m.m, `make check-tools-js` för att se så allt blir installerat och `make install` för att installera alla dependencies. Jag tycker om att testa i docker-kontainrar. Mest för att man får möjligheten att testa kod mot flera olika versioner av node.

Hur väl lyckades du utvärdera TDD-konceptet och vilka är dina reflektioner?

TDD fastnade inte hos mig. Att man ska först skriva testerna och sedan koden. Hos mig går det inte ihop. Jag vill skriva koden först och sedan i slutet skriva testerna. Kanske kommer jag att ändra mig i framtiden men just nu så kommer jag nog skriva koden först och sedan testerna. Det är så jag alltid har gjort.

Berätta om tankarna kring din klient/server applikation och nämn de tekniker du använder.

Jag ska göra en chatt där jag kommer att integrera flera olika API:er (förhoppningsvis). Det första API:et jag kommer att integrera är StackOverflows API. Jag har tänkt lite på hur det ska fungera och jag kom fram till att man kan t.ex. skriva [post]111[/post] för att länka till en post på StackOverflow. Just den där biten med [post] kommer att bytas ut mot en länk till post:en samt byta namn till vad post:en heter. Jag valde att strukturera min kod i 2 olika mappar. En för servern och en för klienten. Servern kör Express och klienten kör Electron. Tillbaka till appen: Förhoppningsvis ska man också kunna logga in (vet inte om detta kommer vara ett krav) och skapa egen profil. Jag tänkte använda StackOverflows inloggnings API, men jag vet inte hur det ska fixas (får tänka på det). Annars får jag skriva min egen variant.

Tillägg:

Eftersom jag utvecklar en Electronapp så fungerar det inte att köra den via docker då det är en desktopapplikation. Därför har jag inte gjort delar av krav 6 i uppgift 2. PS: `make test3` kommer att misslyckas. Den versionen stöds inte.
